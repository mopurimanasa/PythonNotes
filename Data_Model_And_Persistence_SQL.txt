Data Modeling:

-> Data Modeling is the process of designing how data will be stored and related in a database.
-> For example, in an Employee Management System:

Employee Table

Employee_Id		Name	Department_ID
101  			Manasa	10

Department Table:

Department_ID	Department_Name
10				IT

Here:

-> Employee_Id is a Primary Key
-> Department_ID in Employee table is a Foreign Key
-> One department can have many employees(One-to-Many relationships)

Data modeling invloves:

-> Identifying entities (Employee, Department)
-> Defining relationships
-> Choosing primary and foriegn keys
-> Normalization to reduce redundancy

Persistence:

-> Persistence means storing data permanently so it remains available even after the application stops running.

Example:

-> User submits a form.
-> Your Python API receives the request.
-> Data us saved in PostgreSQL.
-> Later, the user can retrieve the same data.

Without persistance:

-> Data would exist only in memory and disappear when the application restarts.

---------------------------------------------------------------------------------------------------------

-> SQL persistence refers to storing and retrieving data using SQL databases such as:

PostgreSQL

Oracle Database

Example:

Insert data -> insert query

Data modeling is the process of designing database structures, defining tables, relationships, primary keys, and foriegn keys to efficiently store and

Normalization to reduce redundancy:

What is Normalization and how to reduce redundancy ?

-> Normalization is the process of organizing data into tables in such a way that redundancy (duplicate data) is reduced and data integrity is improved.

Why do we need Normalization?
Without normalization:
-> Data gets duplicated
-> Updates become difficult
-> Inconsistency happens

❌ Before Normalization (Bad Design)

Single table:
Orders Table
---------------------------------------
order_id | user_name | email | product
---------------------------------------
101      | Manasa    | m@gmail.com | Laptop
102      | Manasa    | m@gmail.com | Phone

Problem:
Manasa and email repeated again and again ❌ (redundancy)

✅ After Normalization (Good Design)

Split into 2 tables:

Users Table
user_id | name   | email
-------------------------
1       | Manasa | m@gmail.com

Orders Table
order_id | user_id | product
-----------------------------
101      | 1       | Laptop
102      | 1       | Phone

✅ Now:

No duplicate user data
Easy to update
Cleaner design

🔥 How Normalization Reduces Redundancy
✅ Step 1: Identify duplicate data
👉 Same values repeated (name, email)
✅ Step 2: Split into separate tables
👉 Move repeating data into a new table
✅ Step 3: Use relationships
👉 Connect tables using Foreign Key

Normalization in databases is divided into different types (forms) to organize data and remove redundancy. Each level has specific rules.

1. First Normal Form (1NF)

👉 Rule:

-> No repeating groups or multiple values in one column
-> Each column should have atomic (single) values

Example:
❌ Not in 1NF:

Student		Subjects
Manasa		Math, Science

✅ In 1NF:

Student		Subjects
Manasa		Math
Manasa		Science
-> One cell = one value

2. Second Normal Form(2NF):

-> Must already be in 1NF
-> Remove partial dependency

Example:

EmployeeID	ProjectID	EmployeeName
1			P1			Manasa
1			P2			Manasa
1			P3			Manasa

-> See how Manasa is repeated 3 times
-> If your name changes, we must update it in 3 places
-> So if we split it:

Employee Table

EmployeeID	EmployeeName
1			Manasa

Employee_Project Table:

EmployeeID	ProjectID
1 			P1
1			P1
1			P1

-> Now "Manasa" is stored only once
-> Remove unnecessary repeated information

3. Third Normal Form (3NF)

-> Must already be in 2 NF
-> Remove Transitive dependency

EmployeeID	EmployeeName	Department
1			Manasa			IT
2			Akhil			IT
3			Ravi			IT

-> See how IT is repeated for every employee?

Employee Table:

EmployeeID	EmployeeName	DeptID
1				Manasa		D1
2				Akhil		D1
3				Ravi		D1

Departmemt Table:

DeptID	Department
D1		IT

-> Now IT is stores only once
-> Move details that belong to another entity into a separate table
