API keys:

What are API Keys?
-> An API key is a unique identifier (like a secret password) used to authenticate requests to an API (Application Programming Interface).
-> An API key is a secret code that allows your application to access another service securely.

Why API keys are used?

✅ Identify who is making the request
✅ Control access to the API
✅ Track usage (rate limiting, billing, logs)
✅ Improve security (only authorized users can access)

When your application calls an API:

GET https://api.example.com/users?api_key=YOUR_API_KEY

Real-world Example

-> When you use Google Maps API, you need an API key
-> When calling AWS services, you use access keys
-> When using OpenAI / ChatGPT API, you pass an API key

Role-Based Access Control (RBAC) and Attribute-Based Access Control (ABAC): 

-> Above two are common security models used to control who can access what in an application or system.

1) Role-Based Access Control (RBAC):
-> RBAC assigns permissions based on roles, and users are assigned those roles.

How it works:

-> Define roles (Admin, Manager, User)
-> Assign permissions to roles
-> Assign roles to users

| Role     | Permissions                  |
| -------- | ---------------------------- |
| Admin    | Create, Read, Update, Delete |
| Manager  | Read, Update                 |
| Employee | Read only                    |

✅ Advantages:

Easy to implement and understand
Good for structured organizations
Faster access checks

❌ Disadvantages:

Not flexible for complex logic
Role explosion (too many roles when conditions increase)

2) Attribute-Based Access Control (ABAC):

-> ABAC controls access using attributes (properties) instead of roles.

Attributes can include:

-> User attributes → department, location, designation
-> Resource attributes → file type, sensitivity
-> Environment attributes → time, IP address
-> Action → read, write, delete

Allow access IF:
user.department == "Finance"
AND resource.type == "Report"
AND access_time < 6 PM

✅ Advantages:

Very flexible and dynamic
Supports real-time decisions
Good for complex, fine-grained control

❌ Disadvantages:

Hard to implement and manage
Requires policy engine

RBAC vs ABAC (Comparison):

| Feature         | RBAC                         | ABAC                         |
| --------------- | ---------------------------- | ---------------------------- |
| Access based on | Roles                        | Attributes (conditions)      |
| Flexibility     | Low                          | High                         |
| Complexity      | Simple                       | Complex                      |
| Scalability     | Medium                       | High                         |
| Use case        | Organizations, internal apps | Cloud, APIs, dynamic systems |

When to use?
✅ Use RBAC when:

You have fixed roles
Simple permission system is enough
Example: Company internal tools


✅ Use ABAC when:

You need fine-grained control
Conditions like time/location/data sensitivity matter
Example: Banking systems, healthcare, AWS IAM policies
--------------------------------------------------------------------------------------------------------------------------
Real-time Example (API Security):

RBAC:
-> If role == "Admin" → allow DELETE /users

ABAC:
If role == "Admin"
AND request.time < 6PM
AND request.ip == company_network
→ allow DELETE /users

--------------------------------------------------------------------------------------------------------------------------

RBAC and ABAC are two access control models.
In RBAC, access is based on roles. Users are assigned roles, and roles have permissions. For example, an Admin can delete users, while a normal user can 
only view.

In ABAC, access is based on attributes like user role, department, time, or location. It evaluates conditions dynamically. For example, a user can access 
data only if they belong to the finance department and are accessing during office hours. RBAC is simple and easy to manage, while ABAC is more flexible 
and suitable for complex, real-time access scenarios.

1. Input Validation

-> When a user sends a request → it enters your API
-> Before using that data → you must validate it

-> Types of Validation:

1. ✅ Syntax validation
Check format

email = "abc@xyz.com" ✅
email = "abc123" ❌
========================================================
2. ✅ Data type validation
age = 25 ✅ (number)
age = "twenty five" ❌
========================================================
3. ✅ Range validation
amount = 1000 ✅
amount = -500 ❌
========================================================
4. ✅ Whitelisting (Best practice)
Allow only accepted values
role ∈ ["user", "admin"]
========================================================
What happens if you don’t validate?

Example → SQL Injection
SELECT * FROM users WHERE id = '1 OR 1=1'
-> Attacker gets all data 💥
-> If validation fails → return error immediately
========================================================

2. Rate Limiting : Rate Limiting is a technique used to control how many requests a user, client, or system can send to an API within a specific time period.

What exactly happens?
System tracks:

-> Number of requests
-> Per user / IP / API key
-> Within a time window
========================================================
Algorithms used:

1. ✅ Fixed Window
100 requests per minute
-> Simple but can burst at boundary
========================================================
2. ✅ Sliding Window (Better)
Smooth control over time
========================================================
3. ✅ Token Bucket (Most used)

Tokens generated at fixed rate
Each request consumes token
No tokens → request rejected
========================================================
Limit: 100 requests/min

User sends:
1–100 → ✅ allowed
101 → ❌ blocked (429 Too Many Requests)
========================================================

3. Encryption in Transit

-> Data is encrypted while it is moving between systems over a network.

Example:
-> Your mobile app sends login details to a server.
-> The data travels over the internet.
-> It is encrypted using protocols like TLS/SSL (HTTPS). so that if someone intercepts the traffic, they cannot read it. 

User -> (HTTPS/TLS encrypted) -> API Server

Common technologies:

-> TLS/SSL
-> HTTPS
-> mTLS
-> SSH

--------------------------------------------------------------------------------------------------------------------------

4. Encryption at Rest

-> Data is encrypted while it is stored on a device, database, disk.

Example:

-> Customer details are stored in a database
-> The database encrypts the data on disk
-> If someone steals the hard drive or gains access to the storage, they cannot read data without the encryption key. 

Common technologies:

-> AES-256
-> Database encryption
-> Disk encryption
-> Cloud storage encryption (such as encrypted object storage)
